# AI Auto-Responder Integration Plan — Telegram Panel + CupidBot API

**Scope:** Add an AI auto-responder to the existing Telegram panel so that every active, logged-in Telegram session can have its private chats monitored and replied to automatically by CupidBot AI, with per-session and per-chat enable/disable controls and persistent per-conversation memory. Instagram is explicitly out of scope.

**Target file:** This plan is written to be executed task-by-task. All paths are relative to the repo root (`/home/jashan/web-pannel`).

---

## 1. Executive Summary

The panel already maintains a live GramJS `TelegramClient` per logged-in Telegram session in `backend/src/services/telegramService.js`. The AI feature will:

1. Attach a persistent `NewMessage` event handler to every connected Telegram session (not only when a browser window is open, as `telegramClientStream.js` does today).
2. Filter incoming messages by per-session and per-chat AI toggles.
3. Maintain a rolling message memory per `(session_id, peer_type, peer_id)`.
4. Call the CupidBot downstream chat-generation API (`POST https://chat-api.cupidbotofm.ai/api/generateChatResponse`) with the conversation context.
5. Send the returned text back through the same GramJS client, via `telegramService.sendMessage`.
6. Expose a management UI under the Telegram panel with session-level master toggles and chat-level overrides.

No Telegram Bot API or Telegram Premium bulk-reply feature is used. Replies go through the logged-in user account's normal MTProto send-message flow.

---

## 2. Requirements Recap

### Must Have
- AI can access **all chats** of **all active logged-in Telegram sessions**.
- AI receives **live incoming messages** for those sessions.
- AI replies automatically after reading each incoming message.
- AI has **persistent per-chat memory** so conversations stay coherent.
- Per-session ON/OFF toggle.
- Per-chat ON/OFF override inside an enabled session.
- Only Telegram accounts/sessions; Instagram excluded.
- Uses the user's own CupidBot API access token and credits.

### Should Have
- Configurable delay/jitter before AI replies to look human.
- Blacklist/whitelist of peer types (user/chat/channel) or specific chats.
- Maximum memory window per chat (e.g., last 50 messages) to control token cost.
- Activity logging and per-session statistics.
- Queue-based processing so CupidBot rate limits don't block the GramJS update loop.
- A dashboard page to view AI status, toggle sessions/chats, and inspect recent AI replies.

### Will Not Do (out of scope)
- Instagram AI auto-responder.
- Telegram Bot API or Telegram Premium bulk-reply features.
- Media generation or sending CupidBot media URLs (text replies only in v1; media can be added later).
- Two-way sync of messages sent by the human operator from outside the panel.

## 5. Back-End Modules

### 5.1 `backend/src/services/cupidbotService.js` — CupidBot API client

Responsibilities:
- Build and POST to `https://chat-api.cupidbotofm.ai/api/generateChatResponse`.
- Read `CUPIDBOT_ACCESS_TOKEN` and optional `CUPIDBOT_ENDPOINT_URL` from env.
- Implement exponential backoff on 429 / 5xx / network errors.
- Normalize the response into `{ text, didConvert, category, rateLimit }`.

```javascript
const { request } = require('undici');
const logger = require('../utils/logger');

const DEFAULT_ENDPOINT = 'https://chat-api.cupidbotofm.ai/api/generateChatResponse';
const ACCESS_TOKEN = process.env.CUPIDBOT_ACCESS_TOKEN || '';
const ENDPOINT_URL = process.env.CUPIDBOT_ENDPOINT_URL || DEFAULT_ENDPOINT;
const MAX_RETRIES = 3;

async function sleep(ms) { return new Promise(r => setTimeout(r, ms)); }

class CupidBotService {
  async generateReply({ accountID, recipient, messages, overrides = {} }) {
    if (!ACCESS_TOKEN) throw new Error('CUPIDBOT_ACCESS_TOKEN is not configured');

    const body = {
      accessToken: ACCESS_TOKEN,
      isAPI: true,
      app: overrides.app || 'telegram',
      brand: overrides.brand || 'cupidbotofm',
      isOF: overrides.isOF !== false,
      accountID: String(accountID),
      recipient: {
        id: String(recipient.id || ''),
        name: recipient.name || '',
        username: recipient.username || '',
        bio: recipient.bio || '',
        location: recipient.location || '',
      },
      messages: messages.map(m => ({
        id: String(m.id),
        timestamp: Math.floor(m.timestamp / 1000),
        msg: m.msg || '',
        isIncoming: !!m.isIncoming,
        medias: (m.medias || []).map(md => ({
          url: md.url || '',
          fileType: md.fileType || 'photo',
          caption: md.caption || '',
          duration: md.duration || 0,
        })),
      })),
      ...overrides,
    };

    let lastErr;
    for (let attempt = 0; attempt < MAX_RETRIES; attempt++) {
      try {
        const res = await request(ENDPOINT_URL, {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify(body),
        });
        const data = await res.body.json();
        if (res.statusCode === 200) {
          const option = data.options?.[0]?.[0];
          return {
            text: option?.msg || '',
            media: option?.media || null,
            didConvert: data.didConvert,
            category: data.category,
            rateLimit: data.rateLimit,
          };
        }
        lastErr = new Error(`CupidBot ${res.statusCode}: ${JSON.stringify(data)}`);
        if (res.statusCode === 429 || res.statusCode >= 500) {
          await sleep(1000 * Math.pow(2, attempt));
          continue;
        }
        throw lastErr;
      } catch (err) {
        lastErr = err;
        if (attempt < MAX_RETRIES - 1) {
          await sleep(1000 * Math.pow(2, attempt));
        }
      }
    }
    throw lastErr;
  }
}

module.exports = new CupidBotService();
```

### 5.2 `backend/src/services/aiMemoryService.js` — memory CRUD

Responsibilities:
- Append a message to `ai_chat_memories` for a given `(sessionId, peerType, peerId)`.
- Trim to the configured `memoryMessageLimit` (default 50).
- Read the current memory window.
- Clear memory for a chat or session.

```javascript
const { pool } = require('../config/database');
const logger = require('../utils/logger');

const DEFAULT_MEMORY_LIMIT = 50;

function _now() { return new Date().toISOString(); }

class AiMemoryService {
  async ensureRow(sessionId, peerType, peerId) {
    await pool.query(
      `INSERT INTO ai_chat_memories (session_id, peer_type, peer_id, messages, message_count)
       VALUES ($1, $2, $3, '[]', 0)
       ON CONFLICT (session_id, peer_type, peer_id) DO NOTHING`,
      [sessionId, peerType, peerId]
    );
  }

  async append(sessionId, peerType, peerId, message, limit = DEFAULT_MEMORY_LIMIT) {
    await this.ensureRow(sessionId, peerType, peerId);
    const incomingAt = message.isIncoming ? _now() : null;
    const outgoingAt = message.isIncoming ? null : _now();
    await pool.query(
      `UPDATE ai_chat_memories
       SET messages = (
         SELECT jsonb_agg(item ORDER BY ord)
         FROM (
           SELECT value AS item, ord
           FROM jsonb_array_elements(messages) WITH ORDINALITY AS t(value, ord)
           ORDER BY ord
           LIMIT GREATEST($5 - 1, 0)
         ) sub
       ) || $4::jsonb,
         message_count = message_count + 1,
         last_incoming_at = COALESCE($6, last_incoming_at),
         last_outgoing_at = COALESCE($7, last_outgoing_at),
         updated_at = NOW()
       WHERE session_id = $1 AND peer_type = $2 AND peer_id = $3`,
      [sessionId, peerType, peerId, JSON.stringify(message), limit, incomingAt, outgoingAt]
    );
  }

  async getMessages(sessionId, peerType, peerId, limit = DEFAULT_MEMORY_LIMIT) {
    const { rows } = await pool.query(
      `SELECT messages FROM ai_chat_memories
       WHERE session_id = $1 AND peer_type = $2 AND peer_id = $3`,
      [sessionId, peerType, peerId]
    );
    if (!rows.length) return [];
    const arr = rows[0].messages || [];
    return arr.slice(-limit);
  }

  async clear(sessionId, peerType, peerId) {
    await pool.query(
      `UPDATE ai_chat_memories
       SET messages = '[]', message_count = 0, updated_at = NOW()
       WHERE session_id = $1 AND peer_type = $2 AND peer_id = $3`,
      [sessionId, peerType, peerId]
    );
  }

  async clearBySession(sessionId) {
    await pool.query(
      `DELETE FROM ai_chat_memories WHERE session_id = $1`,
      [sessionId]
    );
  }
}

module.exports = new AiMemoryService();
```

### 5.3 `backend/src/services/aiSessionManager.js` — listener lifecycle

Responsibilities:
- Maintain an in-memory map of sessionId → unsubscribe function for AI listeners.
- `attach(sessionId)`: register a `NewMessage` handler on the GramJS client if not already attached.
- `detach(sessionId)`: remove the handler.
- Hook into `sessionService.loginSession()` success path and `restoreAllLoggedInSessions()` so listeners start automatically.
- Hook into logout/disconnect so listeners stop.

```javascript
const tgService = require('./telegramService');
const aiChatService = require('./aiChatService');
const logger = require('../utils/logger');

class AiSessionManager {
  constructor() {
    this._handlers = new Map(); // sessionId -> unsubscribe
  }

  async attach(sessionId) {
    const sid = String(sessionId);
    if (this._handlers.has(sid)) return { attached: false };

    try {
      const off = await tgService.addNewMessageHandler(sid, async (event) => {
        try {
          await aiChatService.handleIncomingMessage(sid, event);
        } catch (err) {
          logger.warn(`aiChatService.handleIncomingMessage failed for ${sid}: ${err.message}`);
        }
      });
      this._handlers.set(sid, off);
      logger.info(`AI listener attached for session ${sid}`);
      return { attached: true };
    } catch (err) {
      logger.warn(`AI listener attach failed for ${sid}: ${err.message}`);
      return { attached: false, error: err.message };
    }
  }

  async detach(sessionId) {
    const sid = String(sessionId);
    const off = this._handlers.get(sid);
    if (!off) return { detached: false };
    this._handlers.delete(sid);
    try { await Promise.resolve(off()); } catch (err) {
      logger.debug(`AI listener detach error for ${sid}: ${err.message}`);
    }
    logger.info(`AI listener detached for session ${sid}`);
    return { detached: true };
  }

  async detachAll() {
    const ids = Array.from(this._handlers.keys());
    await Promise.all(ids.map(id => this.detach(id)));
  }
}

module.exports = new AiSessionManager();
```

### 5.4 `backend/src/services/aiChatService.js` — main orchestrator

Responsibilities:
- Decide whether an incoming event should trigger AI reply.
- Fetch settings.
- Update memory.
- Build recipient info.
- Enqueue a BullMQ job for the actual AI call + reply (so the GramJS event loop is never blocked).
- Provide helper methods for manual sync/history fetch.

Key decision logic in `handleIncomingMessage(sessionId, event)`:
1. Ignore outgoing messages (`msg.out === true`) so the AI doesn't reply to itself.
2. Resolve `peerType` and `peerId` from `event.getChat()` / message peer.
3. Check master session setting: `ai_session_settings.enabled === TRUE`.
4. Check per-chat override: if row exists and `enabled === FALSE`, skip.
5. Check peer-type filters in config.
6. Append message to memory.
7. Build lightweight recipient profile from cached dialog entity.
8. Add BullMQ job with `{ sessionId, userId, peerType, peerId, incomingMessage, recipient }`.

```javascript
const { pool } = require('../config/database');
const aiMemoryService = require('./aiMemoryService');
const aiSessionManager = require('./aiSessionManager');
const aiChatQueue = require('../queues/aiChatQueue');
const tcService = require('./telegramClientService');
const logger = require('../utils/logger');

class AiChatService {
  async handleIncomingMessage(sessionId, event) {
    const msg = event.message;
    if (!msg || msg.out) return { handled: false, reason: 'outgoing_or_empty' };

    const chat = await event.getChat().catch(() => null);
    if (!chat) return { handled: false, reason: 'no_chat' };

    const peerType = tcService._peerTypeOf(chat);
    const peerId = tcService._toIdNum(chat.id);
    if (!peerType || peerId == null) return { handled: false, reason: 'bad_peer' };

    const settings = await this.getSessionSettings(sessionId);
    if (!settings.enabled) return { handled: false, reason: 'session_disabled' };

    const chatOverride = await this.getChatSettings(sessionId, peerType, peerId);
    if (chatOverride && chatOverride.enabled === false) {
      return { handled: false, reason: 'chat_disabled' };
    }

    const cfg = settings.config || {};
    const allowedPeerTypes = cfg.allowedPeerTypes || ['user'];
    if (!allowedPeerTypes.includes(peerType)) {
      return { handled: false, reason: 'peer_type_filtered' };
    }
    if (peerType !== 'user' && cfg.allowGroups !== true && cfg.allowChannels !== true) {
      return { handled: false, reason: 'group_channel_filtered' };
    }

    const memoryItem = {
      id: `tg-${msg.id}`,
      telegramMessageId: tcService._toIdNum(msg.id),
      timestamp: Date.now(),
      msg: msg.message || '',
      isIncoming: true,
      medias: [],
    };
    await aiMemoryService.append(sessionId, peerType, peerId, memoryItem, cfg.memoryMessageLimit);

    const sender = await event.getSender().catch(() => null);
    const recipient = {
      id: String(peerId),
      name: tcService._entityTitle(sender || chat),
      username: (sender || chat).username || '',
      bio: '',
      location: '',
    };

    const userId = await this._resolveUserId(sessionId);
    await aiChatQueue.add('generate-reply', {
      sessionId,
      userId,
      peerType,
      peerId,
      incomingMessage: memoryItem,
      recipient,
      config: cfg,
    });

    return { handled: true };
  }

  async getSessionSettings(sessionId) {
    const { rows } = await pool.query(
      `SELECT enabled, config FROM ai_session_settings WHERE session_id = $1`,
      [sessionId]
    );
    if (rows.length) return { enabled: rows[0].enabled, config: rows[0].config || {} };
    return { enabled: false, config: {} };
  }

  async setSessionEnabled(sessionId, userId, enabled, config = {}) {
    await pool.query(
      `INSERT INTO ai_session_settings (session_id, enabled, config, updated_at)
       VALUES ($1, $2, $3, NOW())
       ON CONFLICT (session_id) DO UPDATE
       SET enabled = EXCLUDED.enabled,
           config = EXCLUDED.config,
           updated_at = NOW()`,
      [sessionId, enabled, JSON.stringify(config)]
    );
    if (enabled) {
      await aiSessionManager.attach(sessionId);
    } else {
      await aiSessionManager.detach(sessionId);
    }
    return { sessionId, enabled };
  }

  async getChatSettings(sessionId, peerType, peerId) {
    const { rows } = await pool.query(
      `SELECT enabled, config FROM ai_chat_settings
       WHERE session_id = $1 AND peer_type = $2 AND peer_id = $3`,
      [sessionId, peerType, peerId]
    );
    if (rows.length) return { enabled: rows[0].enabled, config: rows[0].config || {} };
    return null;
  }

  async setChatEnabled(sessionId, userId, peerType, peerId, enabled, config = {}) {
    await pool.query(
      `INSERT INTO ai_session_settings ...`, // ensure session row exists
      [sessionId]
    );
    await pool.query(
      `INSERT INTO ai_chat_settings (session_id, peer_type, peer_id, enabled, config, updated_at)
       VALUES ($1, $2, $3, $4, $5, NOW())
       ON CONFLICT (session_id, peer_type, peer_id) DO UPDATE
       SET enabled = EXCLUDED.enabled,
           config = EXCLUDED.config,
           updated_at = NOW()`,
      [sessionId, peerType, peerId, enabled, JSON.stringify(config)]
    );
    return { sessionId, peerType, peerId, enabled };
  }

  async _resolveUserId(sessionId) {
    const { rows } = await pool.query(
      `SELECT user_id FROM sessions WHERE id = $1`, [sessionId]
    );
    return rows[0]?.user_id;
  }
}

module.exports = new AiChatService();
```

`workers/aiChatWorker.js`:
```javascript
const { Worker } = require('bullmq');
const { getRedisConnection } = require('../config/redis');
const cupidbotService = require('../services/cupidbotService');
const aiMemoryService = require('../services/aiMemoryService');
const tgService = require('../services/telegramService');
const tcService = require('../services/telegramClientService');
const { pool } = require('../config/database');
const logger = require('../utils/logger');

async function sleep(ms) { return new Promise(r => setTimeout(r, ms)); }

async function processGenerateReply(job) {
  const { sessionId, userId, peerType, peerId, incomingMessage, recipient, config } = job.data;
  const logRow = {
    session_id: sessionId,
    peer_type: peerType,
    peer_id: peerId,
    incoming_msg_id: incomingMessage.telegramMessageId,
    status: 'pending',
    request_payload: job.data,
  };

  try {
    const messages = await aiMemoryService.getMessages(sessionId, peerType, peerId, config.memoryMessageLimit);
    const replyDelay = config.replyDelayMs || 3000;
    const jitter = config.replyDelayJitterMs || 2000;
    await sleep(replyDelay + Math.random() * jitter);

    const cupid = await cupidbotService.generateReply({
      accountID: sessionId,
      recipient,
      messages,
      overrides: config.cupidbot || {},
    });

    logRow.response_payload = cupid;

    if (!cupid.text) {
      logRow.status = 'no_reply';
      await _insertLog(logRow);
      return { sent: false, reason: 'empty_reply' };
    }

    const sent = await tgService.sendMessage(sessionId, _buildPeerInput(peerType, peerId), cupid.text, {
      silent: false,
    });

    const outgoingItem = {
      id: `tg-ai-${sent.messageId ?? Date.now()}`,
      telegramMessageId: tcService._toIdNum(sent.messageId ?? sent.id),
      timestamp: Date.now(),
      msg: cupid.text,
      isIncoming: false,
      medias: [],
    };
    await aiMemoryService.append(sessionId, peerType, peerId, outgoingItem, config.memoryMessageLimit);

    logRow.status = 'sent';
    await _insertLog(logRow);
    return { sent: true, messageId: sent.messageId };
  } catch (err) {
    logRow.status = 'failed';
    logRow.error_message = err.message;
    await _insertLog(logRow);
    throw err; // BullMQ will retry based on attempts
  }
}

function _buildPeerInput(peerType, peerId) {
  const { Api } = require('telegram');
  if (peerType === 'user') return new Api.InputPeerUser({ userId: peerId, accessHash: BigInt(0) });
  if (peerType === 'chat') return new Api.InputPeerChat({ chatId: peerId });
  return new Api.InputPeerChannel({ channelId: peerId, accessHash: BigInt(0) });
}

async function _insertLog(row) {
  try {
    await pool.query(
      `INSERT INTO ai_response_logs (session_id, peer_type, peer_id, incoming_msg_id, request_payload, response_payload, status, error_message)
       VALUES ($1,$2,$3,$4,$5,$6,$7,$8)`,
      [row.session_id, row.peer_type, row.peer_id, row.incoming_msg_id,
       JSON.stringify(row.request_payload), JSON.stringify(row.response_payload),
       row.status, row.error_message]
    );
  } catch (err) {
    logger.warn(`ai_response_logs insert failed: ${err.message}`);
  }
}

let worker;
function start() {
  worker = new Worker('ai-chat-jobs', processGenerateReply, {
    connection: getRedisConnection(),
    concurrency: 5,
  });
  worker.on('failed', (job, err) => {
    logger.warn(`ai-chat job ${job.id} failed: ${err.message}`);
  });
}

module.exports = { start };
```

```javascript
}
```

## 9. Key Design Decisions

### Why a BullMQ queue?
The GramJS update dispatcher runs on the same event loop as the MTProto connection. A slow CupidBot API call or a long retry/backoff would block updates for that session. The queue decouples detection from response generation.

### Why per-chat memory in PostgreSQL?
- Conversation history must survive server restarts.
- The panel already uses PostgreSQL as the single source of truth.
- JSONB is flexible enough for the message array and performs well with the GIN/indexed lookup pattern described above.

### Why not use Redis for memory?
Redis is used for queues and locks. Using Postgres for memory keeps the data durable and queryable for the management UI without adding another datastore.

### Why not listen only in `telegramClientStream.js`?
`telegramClientStream.js` only attaches handlers when a browser window is subscribed. The AI must work 24/7 for all logged-in sessions regardless of whether a window is open. Therefore the listener lives in `aiSessionManager.js` and is attached at session connect time.

### How do we avoid the AI replying to itself?
- Incoming `NewMessage` events carry `msg.out`. Outgoing messages are ignored.
- When the AI sends a reply, we append it to memory as an outgoing item.
- The worker does not trigger a new AI cycle for its own outgoing message because it does not go through `NewMessage` from Telegram; it is appended to memory locally.

### Group / channel handling
Default config disables groups and channels (`allowedPeerTypes: ['user']`). Operators can enable them, but the AI will reply as the account itself, which in a channel may require admin rights. The plan keeps the behavior explicit and controlled.

## 11. Security & Operational Considerations

1. **Token storage**: `CUPIDBOT_ACCESS_TOKEN` lives in `backend/.env` only; never send it to the frontend.
2. **Authorization**: every REST endpoint reuses existing `authenticate` + `requireApproved` middleware and checks `sessions.user_id` matches `req.user.id`.
3. **Cross-user isolation**: `ai_chat_*` tables are queried with `session_id`, and session ownership is verified before any read/write.
4. **Rate limiting**: CupidBot returns 429; the worker must backoff exponentially. A global per-minute in-memory counter can be added to shed load before the API call.
5. **Logging**: `ai_response_logs` captures request/response payloads; GDPR/privacy-conscious operators should set a retention job to prune old rows.
6. **Flood waits**: AI-driven sends can hit Telegram `FLOOD_WAIT`. Use `tgService.sendMessage` which already handles retries; add a per-session send queue if needed in v2.
7. **Monitoring**: expose Prometheus-style counters or at least log lines:
   - `ai_session_attached_total`
   - `ai_message_received_total`
   - `ai_reply_sent_total`
   - `ai_reply_failed_total`

## 13. Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| CupidBot API is down or rate-limited | BullMQ retries with exponential backoff; queue preserves jobs; operators can disable AI globally. |
| AI replies in every chat by default | Default session setting is `enabled = FALSE`; explicit opt-in required. |
| AI replies to itself / creates loops | Ignore `msg.out`; append outgoing reply to memory without re-triggering. |
| Telegram flood-wait on AI sends | Use existing `tgService.sendMessage`; add per-session send throttling in v2. |
| Memory grows unbounded | Trim every chat to `memoryMessageLimit`; prune old `ai_response_logs`. |
| Cross-user session access | Reuse existing session authorization in controller/service layers. |
| Instagram scope creep | Only mount routes under `/api/telegram/ai-chat`; only add sidebar item for Telegram platform; capability flag is Telegram-only. |
