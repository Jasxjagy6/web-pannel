3) UPLOAD-SESSION — the path you actually take
--------------------------------------------------------------------------------
The user will give you a `session.json` file (already generated on their
laptop). It looks like:
  {
    "username": "Jashanxjagy7",
    "sessionBlob": {
      "cookies": { ... tough-cookie serialised jar ... },
      "deviceString": "...",
      "deviceId":  "...",
      "uuid":      "...",
      "phoneId":   "...",
      "adid":      "...",
      "build":     "..."
    },
    "proxyUrl": null
  }

To attach it to the panel:
  1. Login at http://localhost:8083/login (admin@example.com / admin123).
  2. Click the platform toggle → Instagram. Verify cinematic overlay plays.
  3. Sidebar → "Upload session"  (route: /instagram/upload-session).
  4. Drag the JSON onto the drop zone, click "Upload N file".
  5. Expect a green "Created session #N for @Jashanxjagy7" result row.
  6. Click "View accounts" → /instagram/sessions.
     The row MUST show Status=Active (green pill) and is_logged_in=true.
  7. Logout of the panel and log back in as admin — the session row must
     STILL be Active. The IG provider rehydrates the cookie jar from the
     encrypted blob in postgres on each request, so no re-login is needed.

If step 6 shows "Inactive" or step 7 shows the session as logged out:
  - Check `sessions.session_data` in postgres is non-null and decrypts.
  - Check backend log for `IG.client.restore` warnings.
  - The decrypt key is in backend/.env (`SESSION_ENCRYPTION_KEY`). If the
    key was rotated since the user generated the blob, you'll see a decrypt
    failure — fix by re-uploading.

5) PRODUCTION-QUALITY GUARDRAILS (don't skip these)
--------------------------------------------------------------------------------
- The panel is multi-tenant. Every IG-side query in the backend must
  WHERE platform='instagram' AND user_id=$1. Don't introduce queries
  that leak rows across users.
- Encrypt session_data on write (see backend/src/utils/crypto.js encrypt()).
  Never store cleartext cookies in postgres.
- Rate-limit & jitter: every IG API call inside scrape loops needs at
  least a 1.5-3s delay with random jitter. IG bans aggressive panels
  fast. This is the single biggest reason scrapers fail.
- Retry backoff: when IG returns 429 / "Please wait a few minutes",
  exponential backoff with cap (max 5min) and surface a friendly
  "Slow down — IG is rate-limiting" toast to the user.
- Never bypass the requireApproved middleware. Subscription gating is
  load-bearing for billing.
- Don't modify Telegram code. Telegram side is locked.
- Don't commit `node_modules`, `.env`, `package-lock.json` mass-rewrites,
  or local test artefacts (`test-plan.md`, `test-report.md` go in PR
  description, not as repo files).

7) GOTCHAS THE PREVIOUS AGENT HIT (so you don't waste time)
--------------------------------------------------------------------------------
- React #31 ("objects are not valid as a React child") will fire if you
  put `{error}` directly in JSX where `error` is an object. Use the
  apiError() helper in frontend/src/utils/apiError.js. It's already
  imported across all IG pages.
- The frontend api/client.js auto-prefixes /<platform>/ on
  platform-agnostic relative URLs based on localStorage('panel_platform').
  So `api.post('/sessions/upload')` from /instagram/* calls
  /api/instagram/sessions/upload automatically. Don't hard-code the
  prefix unless you know what you're doing.
- localStorage('panel_platform') must be 'instagram' BEFORE you call any
  IG API; the platform context sets it on toggle, but if you script
  things in tests, set it manually first.
- Logging-in `Jashanxjagy7` from the VM directly will FAIL with a 400
  "We can send you an email to help you get back into your account" —
  IG's new-device wall. Free SOCKS5 proxies don't help. Use upload-session.
- The IG SDK is `instagram-private-api`; install lives in
  backend/node_modules. The CLI helper resolves it from there.
